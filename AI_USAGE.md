First of all, the whole frontend was created by AI after I finished writing the backend part. Even though I work with Angular in my current company, writing everything from scratch along with proper design would have taken a lot of time, so I let Anti Gravity generate the frontend for me.

There were still some issues which I had to fix myself. The main issue was that it used computed signals for functions that were being used in HTML. These behaved unexpectedly — breakpoints were not being hit and the UI was not updating correctly. I removed them and replaced them with plain functions, which fixed the issue completely.

Another thing I want to mention is that, apart from the `computePersonality` function inside `UserStateService`, nothing in the backend was generated by Anti Gravity. I did take some help from ChatGPT for utility classes, Mongo schemas, and auth middleware. However, there were problems in those as well.

The primary issue was that ChatGPT did not provide a proper interface structure. For maps, it suggested using an object of type `string: number`, which caused problems when I was trying to use them as actual maps in my services. Because of this, I had to delete all question data from the database and reinsert it with a corrected structure.

All core services and the `QARepository` are written by me without AI assistance. I prefer writing services on my own because it helps me understand the flow better. ChatGPT mainly helped with syntax here and there (since it’s hard to remember everything exactly).

The main involvement of ChatGPT was during the initial phase, where I was trying to imagine the flow and validate my thought process. I use it a lot for design discussions and trade-offs.

Below are some of the main prompts I used. I am not including all prompts to keep this file concise.

I initially thought that the question and answer system might use WebSockets, but since this is more of a request-response based flow, APIs make more sense, so I decided to stick with that.

What I was thinking for the flow is like this. First, I can imagine two main APIs for now. One would handle user-related information, and the second would handle the question and answer flow. For the interaction, I was thinking something like getting the next question and then submitting the answer, so these two APIs would be called one after the other.

For state management, I initially thought of storing the state inside the token to save unnecessary latency, but I am not fully sure if that is the best approach. Another idea was to manage state based on module progress. For example, if there are 4 modules and in module 1 the user has already answered 2 questions, then the next time the user is redirected to module 1, we start from question 3. That way, we maintain continuity.

To avoid overly complex logic, I also considered not adding heavy dynamic routing based on user answers. One option I thought of was that if a user shows more inclination towards a particular interest, we can gradually shift them more into that module. After a certain percentage or number of questions in one module, we can continue deeper in that module, while still allowing some jumps if needed.

I am still a bit confused about the deep link part. My current understanding is that if a user revisits a question from a previous or different module, it will still send a request for that question. So again, the same APIs would handle it — first fetching the question and then continuing the flow from there.

For history, I was thinking it can either be stored in memory or in the database. If we store it in the database, we can restore user progress later, but that might introduce some overhead.

I also thought of adding a caching layer. Since multiple users might be accessing similar questions, it may not make sense to hit the database repeatedly for static question data. So caching frequently accessed question data could help improve performance.
